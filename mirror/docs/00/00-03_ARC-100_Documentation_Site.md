## 00-03 ARC-100 Documentation Site

> **What this chapter is.** A description of how the ARC-100 chapters
> in this repository are rendered into a browsable web site. It covers
> the static-site generator (mkdocs), the theme (Material), the
> generator hook that turns [`00-01_ARC-100_Standard_Inventory.md`](00-01_ARC-100_Standard_Inventory.md)
> into the home-page tree, the css/js customisations, the local
> development workflow, and the minimal substitutions required to fork
> the same site to another ARC-100-based project.
>
> **What this chapter is not.** A general mkdocs tutorial (the
> upstream docs at [mkdocs.org](https://www.mkdocs.org/) and
> [squidfunk.github.io/mkdocs-material](https://squidfunk.github.io/mkdocs-material/)
> are authoritative). A reference for site-content rules — those live
> in [`00-00_ARC-100_General.md`](00-00_ARC-100_General.md).
>
> **Why it exists.** Without this chapter, the implicit decisions
> behind the docs site (why the docs root is `docs/`, why the
> generated home lives at `00/index.md`, why the hook lives outside
> `docs_dir`, etc.) are scattered across config-file comments and
> tribal memory. This chapter is the single read for a new contributor
> or a fork target.

### 00-03.1 — Layout at a glance

```text
<repo-root>/
├── mkdocs.arc100.yml                          # site configuration
├── _hooks/
│   └── arc100_master_index.py          # YAML → home-page generator
├── docs/
│   ├── index.md                        # root redirect → 00/index.html
│   └── 00/                             # the ARC-100 indexing root
│       ├── 00-00_ARC-100_General.md
│       ├── 00-01_ARC-100_Standard_Inventory.md      # source of truth (YAML inside)
│       ├── 00-02_ARC-100_Glossary.md
│       ├── 00-03_ARC-100_Documentation_Site.md  # this chapter
│       ├── index.md                    # GENERATED — gitignored
│       └── assets/
│           ├── arc100.css              # custom styling
│           └── arc100.js               # custom behaviour
├── .vscode/
│   └── tasks.json                      # ⌘⇧B launches `mkdocs serve`
├── .gitignore                          # ignores site/ and 00/index.md
└── site/                               # GENERATED — gitignored
```

Two pieces are generated and live in `.gitignore`: the master-index
home at `docs/00/index.md` (regenerated on every mkdocs rebuild from
the YAML in 00-01) and the built static-site output at `site/`
(regenerated by `mkdocs build`). Every other file under `docs/` is
hand-authored prose.

### 00-03.2 — Stack: mkdocs + Material

The site is built with **[mkdocs](https://www.mkdocs.org/)**, a Python
static-site generator that turns markdown files in a directory tree
into a navigable HTML site. The theme is
**[Material for MkDocs](https://squidfunk.github.io/mkdocs-material/)**,
which provides the responsive layout, the left-hand navigation
sidebar, the right-hand TOC sidebar, the search index, and the
dark/light palette toggle.

Why this stack:

- **Markdown is the source-of-truth format** for every chapter, with
  no build step required for an agent or contributor to read the
  files. mkdocs is the rendering layer on top, not a content layer.
- **The `material` theme** is feature-rich (instant navigation,
  sectionable sidebar, search suggest/highlight, code copy buttons)
  without requiring custom theming work.
- **Python ecosystem** matches the ARC-100 hook (also Python), so
  one runtime serves both the site and the index generator.

Install (one-time, system-wide via `pip`):

```bash
pip install mkdocs mkdocs-material pyyaml
```

`pyyaml` is required because the hook parses the YAML block in 00-01
directly.

### 00-03.3 — Configuration: `mkdocs.arc100.yml`

`mkdocs.arc100.yml` at the repo root is the single configuration file for
the site. Key fields:

| Field | Value | Purpose |
| --- | --- | --- |
| `site_name` | `ARC-100 Master Index` | Shown in the top bar and tab title. |
| `docs_dir` | `docs/` | The root of the markdown source tree. |
| `site_dir` | `site` | Where `mkdocs build` writes static HTML. |
| `use_directory_urls` | `false` | Files are served as `<name>.html` rather than `<name>/index.html`, which keeps relative links inside generated content predictable. |
| `hooks` | `_hooks/arc100_master_index.py` | Registers the master-index generator (see §00-03.5). |
| `theme.name` | `material` | The Material theme. |
| `theme.features` | `navigation.instant`, `navigation.indexes`, `navigation.sections`, `navigation.top`, `toc.follow`, `search.suggest`, `search.highlight`, `content.code.copy`, `content.code.annotate` | Material feature flags. `navigation.sections` is what makes the band/book sidebar tiers render as section groups. |
| `theme.palette` | `default` / `slate`; `primary: custom`, `accent: indigo` | Light/dark schemes toggleable in the top bar. `primary: custom` opts out of Material's named-colour scheme so the project-defined `--md-primary-fg-color` (set in `arc100.css` to track `--arc100-base` — see §00-03.5.2) drives the masthead colour. |
| `markdown_extensions` | `admonition`, `attr_list`, `def_list`, `footnotes`, `md_in_html`, `sane_lists`, `tables`, `toc:permalink`, `pymdownx.{details,highlight,inlinehilite,superfences,tilde}` | Standard Material extension set. |
| `plugins` | `search` | Builds the client-side search index. |
| `extra_css` | `assets/arc100/fonts/inter.css`, `assets/arc100/arc100.css`, `assets/arc100/theme-red.css`, `assets/likec4-typography.css` | Loads (in order) Inter @font-face declarations, the parameterized tree-style + Material masthead overrides, the red theme's `--arc100-base` override, and the red-site-only LikeC4 typography. Sitewide so the sidebar treatment applies on every page, not just the generated home. |
| `extra_javascript` | `assets/arc100/arc100.js` | Tree expand/collapse + hover overlay behaviours (single-file IIFE). |

The `nav` is **not** declared in `mkdocs.arc100.yml`. It is generated by the
hook on every build from the YAML index — see §00-03.5.

### 00-03.4 — Site structure

The home page of the served site is the **master index tree**: a
three-level (band → book → chapter) collapsible view of every
chapter, with status filters, expand/collapse-all controls, and a
right-column hover overlay that shows each row's description on mouse
hover. The data behind the tree is the YAML block in
[`00-01_ARC-100_Standard_Inventory.md`](00-01_ARC-100_Standard_Inventory.md) — the same block
that agents extract for machine-readable enumeration.

The home page itself lives at `docs/00/index.md` after generation. To
let visitors land on it from `/`, a small hand-maintained redirect
sits at `docs/index.md`:

```markdown
---
title: ARC-100
hide:
  - navigation
  - toc
---

<meta http-equiv="refresh" content="0; url=00/index.html">
```

Below the home page, every chapter file in `docs/00/` is served at
`/00/<filename>.html`. The hook builds the left-sidebar nav so each
band collapses into its books and each book collapses into its
chapters, with empty books/bands pruned.

### 00-03.5 — Customizations

The site is "stock mkdocs + Material" plus four custom pieces. Each is
deliberately small so that forking ARC-100 only requires substituting
identifiers, not redesigning the system.

#### 00-03.5.1 — Hook: `_hooks/arc100_master_index.py`

This is the most substantive customisation. mkdocs lets a build hook
mutate the config and write generated files in the `on_pre_build`
lifecycle event, before the markdown rendering pipeline runs. The
ARC-100 hook does three things on every build:

1. **Reads** the YAML block between the
   `<!-- ARC-100-INDEX-START -->` and `<!-- ARC-100-INDEX-END -->`
   markers in `<docs_dir>/00/00-01_ARC-100_Standard_Inventory.md`. This is the
   same machine-parseable block that documentation agents extract.
2. **Writes** `<docs_dir>/00/index.md` containing a three-level HTML
   tree (`<div class="band">` → `<div class="book">` → `<div
   class="chapter">`). Active chapters whose `.md` file exists in
   `docs/00/` are rendered as `<a>` rows so the whole row is a
   navigation link; placeholder/draft chapters render as inert
   `<div>` rows with a status badge. Every row carries
   `data-description` and `data-desc-label` attributes that drive
   the right-column hover overlay (see §00-03.5.3).
3. **Overrides** `config['nav']` with a nested structure derived
   from the same YAML. Bands become top-level sidebar sections;
   books become nested sub-sections; active chapters become leaf
   links. Books and bands that contain no rendered chapters are
   pruned so the sidebar never shows orphan headers.

The hook lives **outside** `docs_dir` (at `_hooks/`, not
`docs/_hooks/`). Were it inside the docs tree, mkdocs would copy the
`.py` file into the built `site/` output as a static asset, which is
both noisy and a small information leak. The repo-root location keeps
the hook out of the build product entirely.

The generated `docs/00/index.md` is in `.gitignore` because it is a
derivative of 00-01 and changes on every build. Edits to it would be
overwritten — edit the YAML block in 00-01 instead.

The hook's write to `docs/00/index.md` is **idempotent**: it reads
the existing file content first and skips the write when the new
content is byte-identical. Without this guard, the write would
trigger mkdocs's file watcher (the path is inside `docs_dir`),
re-trigger the hook on the next rebuild, and loop forever spewing
errors during `mkdocs serve`. With the guard, livereload behaves
correctly: the first build creates the file; subsequent builds
rewrite only when the YAML in 00-01 (or the renderer) actually
produces different markdown. **Do not remove this guard** when
editing the hook.

#### 00-03.5.2 — Stylesheet: `assets/arc100/arc100.css`

The custom stylesheet has four responsibilities:

1. **The three-level tree on the home page.** A scoped block under
   `.arc100-tree` defining the band / book / chapter rows. The
   visual hierarchy is driven by a single `--arc100-base` colour
   token and CSS relative-color-syntax (`hsl(from var(--arc100-base)
   h s <L>%)`) to derive every row tone with only the `l` channel
   modulating between rows:

    ```css
    :root {
      --arc100-base:        hsl(0deg 47% 15%);   /* dark red — overridden per theme */
      --arc100-band-bg:     var(--arc100-base);
      --arc100-book-bg:     hsl(from var(--arc100-base) h s 24%);
      --arc100-chapter-bg:  hsl(from var(--arc100-base) h s 32%);
      --arc100-divider:     hsl(from var(--arc100-base) h s 58%);
    }
    ```

    Forking the site to another project is a one-line edit: change
    `--arc100-base` in a per-site `theme-<name>.css` and every
    derived tone follows. The repo ships two per-site theme files:

    - `assets/arc100/theme-red.css` — `hsl(0deg 47% 15%)` (the
      ARC-100 standard's red).
    - `assets/arc100/theme-green.css` — `hsl(120deg 47% 15%)` (the
      ARC-100 Project dogfood's green).

    Each site's `mkdocs.*.yml` loads `arc100.css` first then its own
    `theme-<name>.css` to override `--arc100-base`. Phase 3.1b D3
    centralized the shared assets to the repo-root `assets/arc100/`
    directory (canonical, tracked); the master-index hook copies
    them into each `docs_dir/assets/arc100/` at build time (per-site
    copy, gitignored). Phase 3.1d (V1_STRATEGY §6.8) will introduce
    per-tree LikeC4 toolchain independence; the asset-centralization
    pattern is unaffected.

2. **Sidebar treatment for band/book tiers.** A handful of rules
   under `.md-nav--primary > .md-nav__list > .md-nav__item--section`
   make band headers larger and bolder than book headers so the two
   levels read as distinct tiers in the left sidebar. These rules
   apply on every page (not just the home page), which is why
   `extra_css` loads `arc100.css` sitewide rather than only on the
   generated home.

3. **The right-column hover overlay.** A `position: fixed` panel
   styled in 16px Roboto with a 10% white background. See
   §00-03.5.3.

4. **Material theme palette overrides.** A second `:root` block
   points Material's primary-color variables
   (`--md-primary-fg-color`, `--md-primary-bg-color`, and the
   light/dark variants) at the arc100 palette so the masthead,
   tabs bar, and other primary-coloured surfaces inherit the same
   per-theme base as the band rows. The block also defines
   `--arc100-red-text` (`hsl(from var(--arc100-base) h s 43%)`) — the
   project-wide standard for accent text rendered against a page
   background — and overrides `.md-nav__link--active` and
   `--md-typeset-a-color` to use it. The distinction matters: the
   raw band base is a fill colour (it works under white text on the
   masthead), but it is too dark to read *as* text on either
   scheme. Any accent text in the docs site uses
   `var(--arc100-red-text)`. The override requires `palette.primary:
   custom` in BOTH `mkdocs.arc100.yml` and `mkdocs.arc100project.yml`;
   with any named-colour `primary` value (indigo, red, green, etc.)
   Material's own scheme rules would out-specify these overrides.

#### 00-03.5.3 — Behaviour: `assets/arc100/arc100.js`

The hover overlay is the only behaviour layered onto the home page.
It is a single shared `<div class="arc100-desc-overlay">` element
appended to `<body>` at init time. On `pointermove` over the tree,
the script finds the row under the cursor (`closest('[data-description]')`)
and:

- Writes the row's `data-desc-label` (`"Band 00 thru 09"`, `"Book 01"`,
  or `"Chapter 01-03"`) into the panel's heading span.
- Writes the row's `data-description` (the same text the hook pulled
  from the YAML) into the panel's body span.
- Measures Material's right-hand TOC sidebar (`.md-sidebar--secondary`)
  to position the panel over it, falling back to a viewport-right
  anchor on mobile breakpoints where the sidebar is hidden.
- Sets the panel's `top` to `event.clientY`, then clamps so the panel
  never overflows the viewport bottom or runs above a 12px top inset.

The same JS also drives the home-page **filter dropdown** (Active /
Active+Draft / All), the **expand-all** and **collapse-all** buttons,
and the **click-to-toggle** behaviour on book rows. Each piece is
isolated and short — together the file is under 200 lines.

#### 00-03.5.4 — Root redirect: `docs/index.md`

Because `docs_dir: docs/` and the generated master-index lives at
`docs/00/index.md`, the site root `/` has no natural home page. A
hand-maintained `docs/index.md` containing only a `<meta http-equiv="refresh">`
tag redirects to `/00/index.html` so visitors who land on `/` get
the master index immediately. This file is hand-maintained, not
generated; it tracks no project-specific content.

### 00-03.6 — Local development

The site runs locally via mkdocs' built-in development server, which
watches the docs tree and live-reloads the browser on every change.

#### 00-03.6.1 — Direct invocation

```bash
python3 -m mkdocs serve --dev-addr localhost:8003
```

A few notes:

- The site uses port **8003** by convention (the default `8000` is
  in use by other concurrent mkdocs projects in this environment).
  The port is hard-coded only in the VS Code task and in this
  chapter — both are easy to change for a fork.
- **Livereload covers `.md`, CSS, and JS edits.** mkdocs's file
  watcher fires on changes under `docs_dir`, the hook re-runs (its
  idempotent-write guard described in §00-03.5.1 prevents the
  watcher from looping on the generated `00/index.md`), and the
  browser tab refreshes in place. Editing the YAML in 00-01
  immediately regenerates the home-page tree.
- **`mkdocs.arc100.yml` and `_hooks/*.py` require a server restart.**
  These are loaded once at startup; the watcher can see the edit
  but cannot re-import them. Use the `Docs: Restart MkDocs serve`
  task (§00-03.6.2) for a one-keystroke kill-and-relaunch, or kill
  the listener on port 8003 manually with
  `lsof -ti :8003 | xargs kill` and re-run the serve command. The
  existing browser tab on `localhost:8003` reconnects automatically
  once the server is back.

To check the production build instead (no live-reload, strict
validation):

```bash
python3 -m mkdocs build --strict
```

Output lands in `site/`. The `--strict` flag turns warnings into
errors and catches broken cross-references early.

#### 00-03.6.2 — VS Code: ⌘⇧B / Ctrl⇧B

For day-to-day work the serve command is wired to the VS Code default
build shortcut (⌘⇧B on macOS, Ctrl⇧B on Windows/Linux) via
`.vscode/tasks.json` (outside `docs_dir`, so not linkable from the
served site). The relevant tasks:

| Task | Purpose |
| --- | --- |
| `Docs: MkDocs serve` | Background task that runs `mkdocs serve --livereload --dev-addr localhost:8003` from the workspace root. |
| `Docs: MkDocs serve + open browser` | Depends on the serve task and runs `open http://localhost:8003` so the browser launches once the server is up. |
| `Docs: MkDocs serve ARC-100` | The composite build-shortcut task. Marked `"isDefault": true` under `"group": { "kind": "build" }`, which is what binds it to ⌘⇧B. |
| `Docs: MkDocs build` | One-shot `mkdocs build --clean` (no serve). |
| `Docs: Restart MkDocs serve` | Kills whatever process is listening on 8003 (via `lsof`) and starts a fresh serve. Useful after editing `mkdocs.arc100.yml` or the hook, which are not auto-reloaded. |

The key field is the `group` block on the composite task:

```jsonc
"group": {
  "kind": "build",
  "isDefault": true
}
```

`isDefault: true` is what makes ⌘⇧B run this task without a picker
prompt; without it, ⌘⇧B opens a task selector instead. Only one task
per workspace can hold this flag.

The `background.beginsPattern` / `endsPattern` regexes in the
`problemMatcher` block let VS Code recognise when mkdocs has finished
its initial build, so the dependent "open browser" task waits until
the server is actually ready before opening the URL.

### 00-03.7 — Production build

`mkdocs build` writes a fully-static `site/` directory that can be
served by any plain HTTP server (GitHub Pages, S3, nginx). `site/` is
in `.gitignore` — it is a build artefact and should never be
committed. CI pipelines or hosting integrations should regenerate it
on each deploy.

The generator hook runs identically on `mkdocs build` and `mkdocs
serve`, so the production site and the dev-server preview render
from the same source of truth.

### 00-03.8 — Forking ARC-100 to another project

Adopting ARC-100 in a new project means copying this site
infrastructure essentially verbatim and changing identifiers. The
minimal substitution list:

1. **Rename the brand prefix.** Replace `ARC-100` with the project's
   variant (e.g. `FLOW-100`, `CS-100`) throughout filenames
   (`00-00_ARC-100_General.md` → `00-00_FLOW-100_General.md`), the
   YAML markers (`<!-- ARC-100-INDEX-START/END -->`), the hook's
   `_INDEX_SOURCE` constant, and the human-facing copy in the chapter
   files.
2. **Rename the asset/class prefix.** `arc100.css` / `arc100.js` and
   every `.arc100-*` selector / `--arc100-*` variable inside them.
   This is a single global find-replace.
3. **Set the brand colour.** Change `--arc100-base` in the renamed
   stylesheet. Every derived tone (book, chapter, divider, hover
   accent) follows automatically via `color-mix`.
4. **Pick a port.** Update `--dev-addr localhost:NNNN` in
   `.vscode/tasks.json` (three places — serve, restart, browser-open
   URL) so concurrent forks don't collide on the same port.
5. **Author the per-project chapters.** Keep the ARC-100 books that
   apply, replace placeholders with project-specific content in the
   active-version copy, and add project-specific books in unallocated
   slots per the rules in [`00-00`](00-00_ARC-100_General.md).

What does **not** change between forks: the hook's logic, the
mkdocs.arc100.yml structure (only identifiers), the css/js layout system,
the redirect at `docs/index.md`, and the `.vscode/tasks.json` task
shapes. This chapter, with the identifiers swapped, is itself
generic enough to live unchanged at `00-03` in any ARC-100 fork.
